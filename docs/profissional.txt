FABOT Podcast Studio — Prompt Mestre em 15 Tasks

Contexto Geral
Construir o FABOT Podcast Studio: aplicação web single-page com aparência de software desktop profissional. Fundo branco, tipografia preta, sem gradientes, sem sombras excessivas, sem animações desnecessárias. Visual inspirado em DAWs e IDEs profissionais (Reaper, Cursor, Linear). Stack: React + FastAPI + SQLite + ARQ/Redis + Kokoro local.

TASK 03 — Backend: Ingestão e Limpeza de Texto
Criar services/ingestor.py responsável por extrair texto de qualquer fonte.
Requisitos:

PDF: usar pdfplumber como primário, PyPDF2 como fallback. Após extração, detectar e remover: headers/footers repetidos (comparar primeiras e últimas 3 linhas de cada página), referências bibliográficas (regex para padrões [1], [Author, 2023], notas de rodapé numeradas), numeração de figuras e tabelas.
DOCX: usar python-docx, preservar hierarquia de headings como separadores de seção.
TXT: limpeza simples de encoding e normalização de quebras de linha.
Texto colado: mesma limpeza do TXT.

Validações obrigatórias:

Verificar magic bytes do arquivo, não só extensão.
Rejeitar arquivos acima de 50MB.
Detectar PDFs escaneados (sem texto extraível) e retornar erro descritivo, não silêncio.
Após extração, calcular hash SHA256 do texto limpo para cache de LLM posterior.

Criar utils/text_cleaner.py com função clean_for_tts(text) que: remove emojis, substitui símbolos matemáticos por extenso em pt-BR, expande siglas conhecidas, normaliza números para forma falada ("1200" → "mil e duzentos"), remove URLs.

TASK 04 — Backend: Orquestração LLM
Criar services/llm.py com três providers intercambiáveis via interface comum.
Interface base:
pythonclass LLMProvider:
    async def generate_script(self, text, config) -> dict
    async def health_check() -> bool
Providers:

GroqProvider: modelo llama-3.1-8b-instant. Implementar backoff exponencial (1s, 2s, 4s) em rate limit. Cache de resultado usando hash do texto em SQLite — se mesmo texto + mesma config já foi processado, retorna cached sem chamar API.
GeminiProvider: modelo gemini-1.5-flash. Mesma interface, mesmo cache.
OllamaProvider: endpoint local configurável. Health check antes de usar.

O prompt de geração de roteiro é o coração do sistema. Deve ser um template em arquivo separado prompts/script_template.py com:

System prompt definindo persona de roteirista de podcast educacional brasileiro
Regras explícitas: máximo 3 frases por fala, cada fala máximo 25 palavras, transições naturais entre falantes, uso obrigatório de marcadores [PAUSA_CURTA] e [PAUSA_LONGA] em posições específicas
Instruções de output: retornar JSON estruturado com array de objetos {speaker, text, emotion, pause_after_ms}
Few-shot: 3 exemplos de falas bem formatadas e 2 exemplos de falas ruins para evitar
Instrução de duração: calcular número de falas baseado em palavras_por_minuto=140 e target_duration

Não usar f-string para montar o prompt. Usar Jinja2 para o template do prompt — facilita manutenção futura.

TASK 05 — Backend: Orquestrador TTS
Criar services/tts_orchestrator.py. Este é o módulo mais crítico para qualidade de áudio.
Pipeline por fala:

Receber objeto {speaker, text, emotion, pause_after_ms} do roteiro
Rodar clean_for_tts() no texto
Se texto > 300 chars, quebrar em chunks por sentença (nunca cortar no meio de frase)
Para cada chunk: chamar Kokoro com retry (3 tentativas, backoff 2s/4s/8s)
Se chunk falhar após 3 tentativas: logar com índice exato da fala, continuar com silêncio proporcional
Juntar chunks da mesma fala com crossfade de 40ms
Aplicar pausa correta após a fala baseada em pause_after_ms

Paralelismo controlado:

Processar máximo 3 falas simultaneamente com asyncio.Semaphore(3)
Manter ordem original após processamento paralelo (usar índice)
Logar tempo de cada fala para identificar gargalos

Pausas inteligentes:

Mesma voz continuando: 250ms
Troca de falante: 700ms
Após marcador [PAUSA_CURTA]: 600ms
Após marcador [PAUSA_LONGA]: 1400ms
Início e fim do episódio: 2000ms

Crossfade entre todas as junções: 30ms. Sem isso haverá cliques audíveis entre segmentos.

TASK 06 — Backend: Pós-Produção
Criar services/post_production.py. Único lugar no sistema onde se toca no áudio final consolidado.
Pipeline em ordem fixa:

Normalização RMS para -16 LUFS equivalente (calculado sobre o áudio inteiro, não por segmento)
Compressão dinâmica: threshold -20dB, ratio 2.5:1, attack 20ms, release 150ms
High-pass filter a 80Hz (remove rumble de CPU/ambiente que o Kokoro às vezes gera)
Limiter a -1.0 dBFS
Fade in 1.5s, fade out 3s
Se existir vinheta configurada: overlay nos primeiros segundos com ducking automático (-12dB durante a fala)
Export MP3 192kbps, stereo, 44100Hz

Regra crítica: a normalização acontece UMA VEZ, aqui, no áudio final. Nenhum outro módulo normaliza áudio. audio_utils.py não deve ter normalize().
Gerar também arquivo de log de produção {job_id}_production.json com: duração final, LUFS medido, número de falas, número de falhas de TTS, vozes usadas, tempo total de geração.

TASK 07 — Backend: Worker e Fila
Criar workers/podcast_worker.py usando ARQ com Redis.
Função principal process_podcast_job(ctx, job_id):

Cada etapa do pipeline atualiza job.status e job.progress no banco
Cada etapa é envolta em try/except individual — falha em pós-produção não apaga o roteiro já gerado
Checkpointing: se job já tem script_json preenchido e script_edited=True, pular etapas de LLM e ir direto para TTS
Logar início e fim de cada etapa com timestamp

Mapeamento de progresso:

READING: 0→15%
LLM_PROCESSING: 15→40%
SCRIPT_DONE: 40% (pausa — aguarda possível edição do usuário)
TTS_PROCESSING: 40→85% (atualiza a cada fala gerada: 40 + (falas_geradas/total_falas * 45))
POST_PRODUCTION: 85→95%
DONE: 100%

Criar endpoint SSE /jobs/{id}/stream que faz polling no banco a cada 800ms e envia eventos com {status, progress, current_step, script_json}. Não usar WebSocket — SSE é suficiente e mais simples.

TASK 08 — Backend: API Endpoints
Criar os routers com FastAPI.
routers/upload.py:

POST /upload — recebe multipart/form-data com arquivos + configurações do job. Valida magic bytes. Cria registro no banco. Retorna job_id.
POST /paste — recebe JSON com texto colado + configurações. Cria job. Retorna job_id.

routers/jobs.py:

POST /jobs/{id}/start — enfileira job no ARQ. Muda status para PENDING.
POST /jobs/{id}/start-tts — inicia apenas o TTS (quando usuário editou roteiro). Só funciona se status for SCRIPT_DONE.
GET /jobs/{id}/stream — SSE de progresso
GET /jobs/{id}/result — retorna paths de download quando DONE
GET /jobs/{id}/script — retorna script_json atual
PUT /jobs/{id}/script — salva edição do roteiro, marca script_edited=True
GET /jobs/history — lista todos os jobs com status, duração, data

routers/health.py:

GET /health — verifica: Kokoro (request para /health), Redis (ping), espaço em disco (mínimo 500MB), Ollama se configurado. Retorna JSON com status de cada serviço.

CORS configurado para aceitar apenas localhost:3000 em desenvolvimento.

TASK 13 — Frontend: Estado Global e SSE
Criar store/jobStore.js com Zustand.
Estado:
javascript{

  isConnected: false,      // SSE conectado
  services: {},            // resultado do /health

}
Hook useJobProgress(jobId):

Abre conexão SSE para /jobs/{id}/stream
Ao receber evento: atualiza currentJob no store
Reconecta automaticamente se SSE cair (máximo 5 tentativas, backoff 1s/2s/4s/8s/16s)

Expõe: {status, progress, currentStep, script, audioUrl}

Polling para /health a cada 30 segundos
Resultado disponível no store como services

Toda comunicação HTTP via React Query com retry automático em erros 5xx. Erros 4xx não fazem retry — mostrar mensagem de erro direto.

TASK 15 — Qualidade de Áudio: Regras Finais e Validação
Esta task não é código novo — é revisão e enforcement das regras de qualidade em todo o sistema.
Checklist a verificar em cada arquivo relevante:
tts_orchestrator.py:

 Normalização ausente (apenas post_production.py normaliza)
 Crossfade de 30ms em toda junção de segmentos
 Pausas diferenciadas por contexto (troca de voz vs mesmo falante)
 Retry com backoff em toda chamada ao Kokoro
 Sem abreviações textuais no texto enviado ao Kokoro (vc, pq, tb, etc.)

post_production.py:

 Pipeline na ordem: RMS norm → compressão → high-pass → limiter → fade → vinheta → export
 Export sempre em 192kbps, 44100Hz, stereo
 Geração de _production.json com metadados do episódio

script_builder.py:

 Falas máximo 25 palavras
 Máximo 3 frases por fala
 Marcadores de pausa em JSON, não inline no texto
 Nenhuma abreviação no texto gerado

roteiro_processor.py (herdado):

 Remover dicionário de contrações completamente
 [sorrindo] não injeta palavras no texto — gera pausa de 400ms no JSON
 Marcadores removidos do texto antes de enviar ao Kokoro

Teste de aceitação de qualidade: gerar episódio de 5 minutos e verificar: sem cliques audíveis nas junções, volume consistente entre host e co-host, pausas naturais entre falantes, sem artefatos no início/fim de cada fala, LUFS final entre -18 e -14.


Ordem de implementação: Tasks 01→02→03→07→08 (backend funcional mínimo) → 09→10→13 (frontend mínimo com upload) → 04→05→06 (pipeline de qualidade) → 11→12 (editor e player) → 14 (Docker) → 15 (auditoria final). Nunca pular a Task 15 — é a diferença entre TTS amador e podcast profissional.
